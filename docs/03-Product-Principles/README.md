# 1. Introdução

Os Product Principles representam a filosofia central do Forge.

Toda decisão relacionada ao produto deve respeitar estes princípios.

Nenhuma funcionalidade é mais importante do que a identidade do produto.

Se uma nova ideia violar estes princípios, ela deverá ser repensada.

---

# 2. Nossa Filosofia

Forge não existe para adicionar funcionalidades.

Forge existe para ajudar pessoas a evoluírem.

Cada botão.

Cada tela.

Cada animação.

Cada gráfico.

Cada linha de código.

Tudo deve contribuir para este objetivo.

---

# 3. Os 20 Princípios

---

## 1. Evolução acima de tudo.

Toda funcionalidade deve incentivar evolução real.

Nunca distração.

---

## 2. Simplicidade vence complexidade.

Se algo puder ser feito de forma simples, essa será a escolha.

---

## 3. Dados precisam gerar entendimento.

Não basta mostrar números.

O usuário deve entender o que eles significam.

---

## 4. A tecnologia deve desaparecer.

O usuário nunca deve perceber a complexidade por trás do sistema.

Tudo deve parecer natural.

---

## 5. O usuário é dono dos seus dados.

Os dados pertencem ao usuário.

Forge apenas os protege.

---

## 6. Offline sempre que possível.

O usuário nunca deve deixar de registrar um treino por falta de internet.

---

## 7. Performance é uma funcionalidade.

Rapidez faz parte da experiência.

Uma tela lenta é considerada um defeito.

---

## 8. Consistência gera confiança.

Todos os componentes devem seguir os mesmos padrões.

---

## 9. Cada clique deve ter um propósito.

Nada será colocado apenas para "encher espaço".

---

## 10. Interfaces devem ensinar.

A interface deve ajudar o usuário a aprender naturalmente.

---

## 11. IA complementa.

Nunca substitui o pensamento do usuário.

Ela orienta.

Não decide.

---

## 12. Gamificação deve motivar.

Nunca manipular.

Nunca gerar ansiedade.

Nunca criar vício.

---

## 13. A comunidade deve construir.

Nunca competir de forma tóxica.

---

## 14. Acessibilidade é obrigatória.

O Forge deve ser utilizável pelo maior número possível de pessoas.

---

## 15. Segurança vem primeiro.

Nenhuma funcionalidade justifica comprometer a privacidade.

---

## 16. Crescer sem perder qualidade.

Adicionar funcionalidades nunca deve reduzir a qualidade do sistema.

---

## 17. Design transmite confiança.

Visual bonito não é suficiente.

A interface deve parecer confiável.

---

## 18. Código limpo é investimento.

Cada linha escrita hoje deve facilitar o trabalho de amanhã.

---

## 19. Toda funcionalidade deve responder:

Por quê?

Para quem?

Qual problema resolve?

Como melhora a evolução?

Se não houver resposta clara, ela não deve existir.

---

## 20. O Forge nunca estará pronto.

Sempre haverá espaço para melhorar.

A melhoria contínua faz parte da identidade da plataforma.

---

# 4. Como Tomamos Decisões

Sempre que surgir uma nova ideia, responder:

✔ Resolve um problema real?

✔ Está alinhada com a missão?

✔ Mantém a simplicidade?

✔ É escalável?

✔ É intuitiva?

✔ Respeita a privacidade?

✔ Pode funcionar offline?

✔ Ajuda na evolução do usuário?

Se a maioria das respostas for "não", a ideia deve ser reavaliada.

---

# 5. Checklist para Pull Requests

Antes de aprovar qualquer alteração:

- [ ] O código segue os padrões?
- [ ] O componente é reutilizável?
- [ ] Está documentado?
- [ ] Mantém consistência visual?
- [ ] Não prejudica performance?
- [ ] Não aumenta complexidade desnecessária?
- [ ] Respeita acessibilidade?
- [ ] Respeita os princípios do Forge?

---

# 6. Revisão dos Princípios

Os princípios podem evoluir.

Entretanto, qualquer alteração deverá preservar a missão central:

**Ajudar pessoas a evoluírem de forma consistente utilizando tecnologia.**
